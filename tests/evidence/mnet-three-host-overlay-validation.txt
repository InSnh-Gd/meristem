Issue found during validation
- Before the final fix, repeated node-agent reconciliation ran `wg setconf` even when the rendered config hash did not change.
- In the real relay topology this interacted with WireGuard endpoint roaming and caused unstable overlay ICMP:
  - Stem -> Leaf: 100 packets transmitted, 82 received, 18% packet loss after the first temporary OUTPUT redirect.
  - Leaf -> Stem: 100 packets transmitted, 87 received, 13% packet loss after the first temporary OUTPUT redirect.
  - A tried POSTROUTING SNAT stabilization broke traffic and was reverted immediately.

Fix loaded on Stem and Leaf
- services/node-agent/src/node-agent-local-apply.ts now reads the previous local overlay state configHash.
- If the rendered config hash is unchanged and the WireGuard interface already exists, reconciliation skips `wg setconf` while still refreshing address, link state, and routes.
- The state `configHash` is persisted only after WireGuard apply succeeds, so a partial failure after interface creation cannot make the next reconciliation falsely skip `wg setconf`.
- Regression test added: `node-agent local overlay apply > does not reset WireGuard when the rendered config hash is unchanged`.
- Regression test added: `node-agent local overlay apply > retries WireGuard setconf after a partial apply failure`.
- Stem and Leaf were synced with the guarded local apply code and node-agents restarted.

Temporary live-host validation limitation
- This run used reversible host-local netfilter plumbing while validating the real-host tunnel path.
- The exact host rule is intentionally omitted from repo evidence because it is not a supported M-Net production mechanism.
- Slice A follow-up evidence (`tests/evidence/mnet-routing-no-host-redirect.txt`) defines the product boundary: node-agent renders the declared local wstunnel UDP sidecar endpoint and committed node-agent apply logic must not depend on ad hoc host firewall/netfilter redirect commands.
