# ADR-F02: 架构组织原则

## 状态

Accepted

## 上下文

Meristem 的产品边界要求 Core 保持微内核，复杂能力进入明确的 M-* 功能域或微服务。为了避免实现形态与产品能力混淆、防止 Core 膨胀为业务单体，需要一组明确的架构组织原则。

关键问题：

- 如何表达服务边界，避免 Core 成为隐式耦合中心？
- 如何防止“插件优先”或“平台单体”的架构漂移？
- 身份、密钥、性能等横切关注点应放在哪里？

## 决策

### 轻量微服务

Meristem 采用 Monorepo 内的轻量现代微服务。默认不引入 Service Mesh、gRPC everywhere、每服务独立数据库或重型编排平台。

### Core 微内核

Core 是微内核，负责：

- bootstrap 与基础配置加载
- 基础身份入口
- 服务生命周期入口
- Elysia app composition
- REST + OpenAPI
- 内部 Eden 契约聚合
- M-CLI 入口
- 安全模式
- 最小日志入口与最小策略入口
- secretRef 管理入口
- 节点注册入口
- Core 自身健康检查

Core 禁止直接拥有完整的 M-Net 策略算法、完整日志分析、完整审计查询、完整风险模型、完整 LLM 分析流程、OpenSearch 读模型实现或复杂业务微服务逻辑。

### 微服务是实现形态，不是一级模块

不创建 M-Services 作为一级模块。微服务是 M-* 功能域的主要实现形态。这一原则防止实现机制被误认为产品能力。

### M-Extension 命名与定位

M-Plugin 已重命名为 M-Extension。Extension 是补充扩展机制，仅在微服务无法满足扩展目标时使用。Extension 默认低权限，其权限、manifest 和生命周期必须保持显式。

### 不设 M-Perf 模块

性能是各模块内部实现策略，不设立 M-Perf 一级模块。跨切面的性能工具可以存在于共享包中，但不能成为顶层产品域。

### Identity 属于 Core

不创建 M-Identity 一级模块。基础身份属于 Core，授权和风险逻辑属于 M-Policy。

### Secrets 分散到 Core / M-Policy / M-Log

不创建 M-Secret 一级模块。Core 拥有 secretRef 和密钥管理入口；M-Policy 授权密钥访问；M-Log 审计密钥操作。实现中必须防止密钥值进入日志、OpenSearch、LLM 提示词或 UI 错误信息。

### M-Policy 起点

M-Policy 从 RBAC 开始。后续再分层操作危险等级、置信度、置疑度、LLM 解释和多元决策流程。v0 的可测试权限基础不能绕过 RBAC 或 Audit Log。

## 结果

- Core 保持小而聚焦，复杂能力下沉到 M-* 域。
- 能力边界保持以产品域为导向，而不是以部署形态为导向。
- 命名上的转变降低了插件优先架构漂移的风险。
- 身份、密钥、性能等横切关注点获得明确 owner，避免过早拆分为独立模块。

## 重访条件

- 如果部署规模或组织边界要求更重的微服务基础设施，且成本已被记录。
- 如果某个能力无法安全地在 Core 外实现，同时仍保持微内核边界。
- 如果 Extension 生态系统成为有意为之的主要产品特性。
- 如果身份、密钥或性能成长为独立的产品域，拥有超出 Core 应承受范围的生命周期和契约。
- 当 RBAC 已实现并可以安全地叠加风险原语之后。
